# Phase Five: Decoder Shape

This note records the current model of the `W[J]` decoder path derived from targeted state tracing in `helper_21_a.lua`.

## High-confidence picture

`W[J]` is not a simple string-table lookup helper.

It behaves like a seeded decoder with:

- a cache table keyed by the second argument
- a per-call queue of generated byte offsets
- a seeded PRNG/refill helper
- a final byte-to-character table

## `W[J]` arguments

From the traced state for wrapper state `15004855`, the closure uses:

- arg 1: the binary blob to decode
- arg 2: the seed / cache key

The first action is effectively:

- `cache = W[v[1]]`
- `cached = cache[seed]`
- if cached exists, return the cached decoded string
- otherwise start a seeded decode path

One important correction from this pass:

- `W[J]` most likely does **not** return the decoded string itself
- it appears to populate `cache[seed]` and then return the seed key
- the caller then resolves the actual plaintext through `W[b][returned_key]`

That matches the observed call pattern throughout the loader branches:

- `key = W[J](blob, seed)`
- `value = W[b][key]`

and also matches the traced miss path, which writes the accumulating decoded output directly into `cache[seed]`.

## Cache-miss initialization

The miss path initializes:

- `queue = {}`
- `state45 = seed % 2^45`
- `state257 = seed % 255 + 2`
- `cache[seed] = ""`

It also computes `string.len(blob)` and then enters a per-byte loop.

## Per-byte decode loop

For each byte in the blob, the traced logic does the equivalent of:

1. `byte = string.byte(blob, i)`
2. `rand = next_generated_value()`
3. `decoded = (byte + rand + 101) % 256`
4. append `byte_table[decoded + 1]` to the cached output string

The traced state makes it clear that:

- the constant `101` is real
- the byte table is external to the blob and reused
- the decoded string is accumulated directly in the cache table entry for `seed`
- the return path likely hands back the seed key rather than the plaintext itself

## Queue helper

The per-byte helper used by `W[J]` is a second closure that behaves like this:

- if the queue is empty, refill it
- otherwise `table.remove(queue)` to pop the next generated value

The refill path is now strongly characterized:

### 45-bit state

It updates a 45-bit state using:

- `state45 = (state45 * 177 + 5746771741299) % 2^45`

### 257-state

It updates a small-state value using:

- `state257 = (state257 * 80) % 257`

and skips the `1` state by advancing again when needed.

### Four-byte refill

The refill path derives four queued byte values from the two states.

The exact traced shape is:

- use `state257 % 32`
- use `floor(state257 / 32)` through arithmetic form
- derive a power-of-two divisor
- derive a 32-bit window from the 45-bit state
- split that 32-bit value into four byte-sized values
- store those four values in the queue

The fetch path then pops from that queue using `table.remove(...)`.

## What this means

This is enough to say:

- the hidden strings are not random per-call junk
- they are reproducible offline from `(blob, seed)` plus the byte table
- the remaining unknowns are concentrated in the final byte table and a small amount of helper plumbing
- `W[b]` is not an unrelated second decoder; it is the cache/proxy surface that exposes the decoded value by key

## Practical next step

## Tracer-backed update

The new state tracer confirms a pre-loader setup path that is highly relevant to the decoder captures later used by `W[J]`.

### Confirmed helper setup

One initialization state wires up the exact helpers we suspected:

- `math.floor`
- `math.random`
- `table.remove`
- `string.char`

and stores them in slot-backed locals that stay live into the later bootstrap work.

### Confirmed `h` seed list

The same setup path seeds a local `h` list and a later loop writes:

- `h[x] = x`

with:

- `x` starting at `0`
- `E = 1`
- `m = 256`

which is the shape of a `for i = 1, 256 do h[i] = i end` initializer.

### Confirmed `W[o]` staging table

Another loop on the same path does the equivalent of:

1. `idx = math.random(1, #h)`
2. `x = table.remove(h, idx)`
3. `W[o][x] = string.char(x - 1)`

This strongly suggests that at least one lifetime of local `o` is a staged byte table built from the canonical byte range.

### Important contradiction

That setup appears to make the final byte table an identity-style `index -> char(index - 1)` table.

However, the current offline decoder still does **not** yield readable Roblox-style plaintext for the hidden `GetService(...)` names or the short `Instance[...]` method names.

So one of these must still be true:

- the captured `o` used by `W[J]` is replaced or mutated again after the visible staging loop
- the current backtrace has only recovered one phase of `o`'s lifetime, not the final one used at capture time
- there is still a wrong assumption in the seeded byte-stream math despite the traced leaf matching the current model closely
- or there is an additional transform on the hidden strings that has not been identified yet

### Ruled-out mismatch

The stage-2 pool dump matches the inline `H[index]=hex:...` annotations for the blobs used in `W[J](blob, seed)`, so the blob source itself is not currently suspected.

The next solve step is not “guess more strings.”

It is:

1. trace whether `o` is reassigned or mutated again on the path from the helper setup states into the bootstrap leaf
2. keep validating the seeded byte-stream math against the traced queue/refill states
3. only then return to bulk-decoding the hidden `(blob, seed)` pairs

Once that contradiction is resolved, the hidden `GetService(...)` names and `HttpGet(...)` targets should become decodable in bulk.
